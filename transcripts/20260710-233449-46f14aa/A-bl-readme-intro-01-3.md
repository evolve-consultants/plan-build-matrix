# A friendlier README intro

Here's a template you can adapt:

---

## What is [Project Name]?

[Project Name] helps you [core benefit in plain language]. Instead of [old painful way], you can now [what becomes possible].

**Quick example:** [One concrete scenario showing the before/after]

## Why does this matter?

- **[Specific pain point]** — [How this solves it]
- **[Specific pain point]** — [How this solves it]
- **[Specific pain point]** — [How this solves it]

## Get started in 30 seconds

```bash
[Most minimal working example possible]
```

That's it! You now have [what they can do].

---

## Tips for filling this in:

- **Lead with outcome, not features.** ("Deploy faster" beats "supports containerization")
- **Use "you" language.** Makes it feel less formal
- **Show, don't tell.** One working code snippet beats three paragraphs
- **Cut ruthlessly.** If it doesn't answer "what is this?" or "why should I care?", remove it
- **Save the details.** Longer explanations belong later in the README
